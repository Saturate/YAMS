## Why HUSK

Most AI memory tools are local-only — memories live on one machine and disappear when you switch laptops. HUSK is a self-hosted server that all your machines and teammates connect to.

- **Cross-machine by default.** Run HUSK on a server or in Docker, point your AI tools at it from anywhere. No sync, no export/import — it's just always there.
- **Team-ready.** Create users, generate API keys, invite a GitHub org. Not just "works for me on my laptop."
- **Scoped retention.** Memories belong to a session, project, or global scope with independent TTL. Session context expires, architectural decisions stick around.
- **Token-budgeted retrieval.** Ask for memories with a token limit — HUSK returns as many as fit. No context window blowup.
- **Semantic duplicate detection.** Configurable similarity threshold so the same insight phrased differently doesn't pile up.
- **MCP native.** Works with Claude Code, and any MCP-compatible tool out of the box.

If you work on one machine and don't need multi-user, simpler tools like [claude-diary](https://github.com/rlancemartin/claude-diary) or claude-tandem might be enough. If you want a managed cloud service, [Supermemory](https://github.com/supermemoryai/supermemory) or [Khoj](https://github.com/khoj-ai/khoj) are worth a look.

## Approaches to AI memory

Not all memory systems work the same way. Understanding the approaches helps explain why different tools make different trade-offs.

### Vector / semantic search

Store memories as text, embed them into vectors, find similar ones by meaning. This is what HUSK does today with Qdrant and Ollama.

**Strengths:** Natural language queries, fuzzy matching, finds relevant context even when wording differs.
**Weaknesses:** No explicit relationships between memories. "Find me things *about* auth" works great, "what *caused* the auth rewrite?" doesn't.

### Knowledge graphs

Model memories as nodes with typed edges between them: decisions, consequences, contradictions, temporal sequences.

**Strengths:** Relationship queries, causal chains, "what changed since last week?", contradiction detection.
**Weaknesses:** More complex infrastructure (Neo4j, or graph-in-SQLite), requires entity extraction to build the graph.

### File-based / CLAUDE.md injection

Write learnings to markdown files. The AI reads them at session start. No search, no database.

**Strengths:** Zero dependencies, dead simple, works with any AI tool that reads project files.
**Weaknesses:** Doesn't scale. No search, no scoping, no dedup. You're limited to what fits in a flat file.

### Content-addressed storage

Hash and deduplicate file content at the block level, with search layered on top. More "smart git" than memory system.

**Strengths:** Efficient storage of actual files, snapshot diffs, no duplicate content.
**Weaknesses:** Optimized for files and code, not for discrete memories or observations.

### Hybrid approaches

Combine vector search with knowledge graphs and/or full-text search. Use the right retrieval method for the query type.

HUSK currently uses vector search. Adding a graph layer on top (not replacing vectors) would enable relationship-aware retrieval alongside semantic similarity. For example: vector search finds "we switched to session cookies" and a graph edge connects it to "XSS vulnerability in the JWT flow" as the cause. This is on the [roadmap](#whats-next-for-husk).

## Agent and chatbot memory platforms

These tools target a different use case: giving AI agents and chatbots memory about their *end users*. "Remember that this customer prefers dark roast coffee" rather than "remember that this codebase uses the repository pattern."

They're worth knowing about, but solve a different problem than HUSK.

| Platform | What it's for | Open source | Approach |
|---|---|---|---|
| [Mem0](https://mem0.ai/) | Memory API for AI apps. User/session/agent scopes for personalizing chatbots and agents. 48.5k GitHub stars, $24M Series A. | Yes | Vector + graph |
| [Zep](https://www.getzep.com/) | Enterprise context platform. Temporal knowledge graphs that track how customer preferences evolve. Graphiti engine (23.2k stars) is open source. | Partial | Temporal knowledge graph |
| [Letta](https://www.letta.com/) | Stateful agent platform (formerly MemGPT). Agents self-manage their own memory using an OS-inspired hierarchy: core memory (RAM) vs archival memory (disk). 20.3k stars. | Yes | OS-style memory tiers |
| [Cognee](https://www.cognee.ai/) | Knowledge engine with ECL pipeline (Extract, Cognify, Load). Self-improving memory that prunes stale nodes and reweights edges. 12.9k stars. | Yes | Knowledge graph + vector |

## Claude's native memory

Claude has its own memory systems that are separate from HUSK. Understanding how they work helps explain where HUSK fits in.

### Claude.ai memory

The consumer product (claude.ai) remembers preferences and context across conversations. Processed in daily synthesis cycles. No API, UI-only management. Not relevant for developer tooling.

### Claude Code auto memory

Claude Code writes notes to `~/.claude/projects/*/memory/MEMORY.md` as it works. Plain markdown, machine-local, no sync, no search. HUSK fills the gaps: cross-machine sync, semantic search, scoped retention, and multi-user support.

### Claude Memory API

The most interesting integration point. The [Memory API](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/memory-tool) (`memory_20250818`) is a client-side tool where Claude emits file operations (`create`, `view`, `str_replace`) and *your application* handles the storage. Anthropic stores nothing.

HUSK could act as the storage backend for this tool, adding vector indexing, semantic dedup, scoping, and TTL on top of Claude's native memory operations. This is on the roadmap.
